# 모바일 추첨 무대 UI 재설계

- 작성일: 2026-07-15
- 상태: Approved
- 승인일: 2026-07-15
- 승인자: 사용자
- 최종 위험등급: Standard — 화면 구조와 상호작용 표현을 바꾸지만, 번호 생성 규칙·저장 데이터·외부 API·배포 설정은 변경하지 않는다. 사용자가 이 등급과 범위를 확인했다.
- 확정 설계: docs/design/mobile-draw-stage-ui.md

## 목적

로또 번호 생성기의 첫 화면을 작은 화면에서도 즉시 사용할 수 있는 **일반 웹 도구형 화면**으로 재구성한다. 기존의 남색·금색·공·추첨기 정체성은 포인트로만 유지하고, 익숙한 헤더–생성 영역–결과–정보 섹션 구조 안에서 `번호 뽑기`와 직전 결과 확인을 우선하여 320px 폭에서도 가로 넘침이나 겹침 없이 완결된 흐름을 제공한다.

## 요구사항

- 모바일 우선으로 헤더, 짧은 안내, 최신 당첨정보, 모드 선택, 횟수 선택, `번호 뽑기`, 결과 용지의 시각적 우선순위를 다시 구성한다.
- 기존 `순수 랜덤`, `역대 단골 번호`, `행운수 고정`과 1~5게임 선택의 동작과 라벨 의미를 유지한다.
- 추첨 오버레이 확인 뒤에만 결과 용지에 반영하는 현재 결과 확정 흐름, `prefers-reduced-motion` 즉시 결과 흐름, 소리 켜기/끄기 저장 동작을 유지한다.
- 최신 당첨번호, 확률 비예측 고지, 데이터·출처 고지 및 기존 통계/당첨금 섹션은 삭제하거나 의미를 바꾸지 않는다.
- 기존 CSS와 React 컴포넌트 경계를 우선 재사용한다. 새 디자인 시스템·UI 라이브러리·이미지 의존성은 추가하지 않는다.

## 사람 확인 방향 — 기본 웹 구조 + 로또 포인트

- 구조: `헤더 → 짧은 소개/최신 당첨번호 → 번호 생성 영역 → 내 결과 → 당첨금액/통계 → 출처`의 일반적인 웹 흐름을 사용한다. 각 영역의 목적을 첫눈에 구분할 수 있게 제목과 여백을 둔다.
- 분위기: 과도한 방송 세트·네온·게임 화면 대신, 읽기 쉬운 어두운 남색 바탕과 밝은 표면을 기본으로 한다. 금색과 번호 공의 색은 로고, 핵심 버튼, 선택 상태, 결과의 강조에만 사용한다.
- 우선순위: 생성 영역은 첫 화면의 주인공으로 두되, 큰 장식 추첨기는 보조 시각 요소로 축소하거나 제거한다. `번호 뽑기`는 가장 분명한 단일 행동으로 만든다.
- 형태: 카드·테두리·그림자는 꼭 필요한 그룹 구분에만 쓰며, 선택 상태는 색만이 아닌 테두리·명도·버튼 텍스트로도 구별한다.
- 반응형: 폭이 줄면 공·칩·번호판의 열 수와 간격을 단계적으로 줄이며, 콘텐츠를 가로로 자르거나 수평 스크롤시키지 않는다.

## 완료 조건

이 spec의 완료 조건은 설계와 독립적이다. 각 항목은 데스크톱 브라우저와 실제 모바일 폭의 렌더링/상호작용으로 확인한다.

- [ ] **AC-001 — 모바일 완결성:** 320px와 390px CSS 폭에서 본문, 최신 당첨번호, 모드칩, 횟수 선택, `번호 뽑기`, 결과 용지가 수평 스크롤 없이 보이고 조작할 수 있다. 헤더·소리 토글·보조 문구가 서로 겹치거나 잘리지 않는다.
- [ ] **AC-002 — 생성 흐름 보존:** 세 가지 생성 모드와 ×1~×5 선택은 기존과 같은 결과 생성 함수를 호출한다. `행운수 고정`에서 1~45 번호를 선택/해제하고 최대 5개 제한 및 전체 해제가 정상 동작한다.
- [ ] **AC-003 — 결과 확정 흐름 보존:** 일반 모션 환경에서는 추첨 오버레이의 확인 이전에 새 결과 용지가 본문에 나타나지 않고, 확인 뒤 결과가 표시된다. 모션 감소 환경에서는 기존처럼 오버레이 없이 결과가 즉시 표시된다.
- [ ] **AC-004 — 정보의 의미 보존:** 최신 회차의 본번호 6개와 보너스 번호, `역대 단골 번호`/`행운수 고정`이 예측이 아닌 재미 요소라는 고지, 데이터 없음 폴백 문구, 출처·비공식 고지가 계속 노출되며 문구의 의미가 약화되지 않는다.
- [ ] **AC-005 — 시각적 위계:** 일반 웹처럼 헤더·생성·결과·정보 섹션이 구분되고, 첫 화면에서 사용자는 브랜드/회차 정보보다 먼저 또는 같은 시선 흐름에서 현재 모드, `번호 뽑기`, 선택 게임 수를 식별할 수 있다. 결과가 있을 때는 생성 제어부와 결과 용지가 한 흐름으로 인접하게 보인다.
- [ ] **AC-006 — 접근성:** 모든 기존 버튼은 키보드로 도달·작동하며 포커스 표시가 식별된다. 선택 상태와 비활성 상태를 색만으로 전달하지 않고, 기존의 버튼 텍스트·`aria-pressed`·`disabled` 의미를 훼손하지 않는다.
- [ ] **AC-007 — 넓은 화면 회귀 방지:** 768px와 1280px 폭에서도 콘텐츠가 과도하게 늘어나지 않고, 최신 당첨번호와 생성 제어부가 읽기 쉬운 간격을 유지한다.
- [ ] **AC-008 — 기술 검증:** 관련 단위/컴포넌트 테스트, lint, typecheck, production build가 통과한다. 변경 전후의 320px·390px·768px·1280px 브라우저 확인 결과와 실행하지 못한 검증은 별도로 기록한다.

## 정상·오류·경계 시나리오

| 구분 | 시나리오 | 기대 결과 |
| --- | --- | --- |
| 정상 | 순수 랜덤 ×1을 선택하고 번호를 뽑는다 | 기존 추첨 연출 또는 모션 감소 결과 경로를 거쳐 6개 번호가 결과 용지에 표시된다. |
| 정상 | 행운수 고정에서 0~5개 번호를 고르고 ×5를 선택한다 | 각 게임이 기존 고정 번호 규칙을 유지하며, 5개 초과 선택은 막힌다. |
| 오류/빈 데이터 | 과거 데이터가 비어 있는 상태에서 역대 단골 번호를 고른다 | 기존 랜덤 폴백과 `데이터 없음` 안내가 유지된다. |
| 경계 | 320px 폭에서 최신 회차 본번호 6개와 보너스 번호를 표시한다 | 번호 공이 줄바꿈/축소/재배치되어 가로 넘침 없이 모두 식별된다. |
| 경계 | 320px 폭에서 1~45 번호판을 연다 | 번호 버튼이 잘리거나 화면 밖으로 밀리지 않으며, 선택·해제·전체 해제가 가능하다. |
| 경계 | `prefers-reduced-motion: reduce` 환경에서 번호를 뽑는다 | 애니메이션을 강제하지 않고 결과가 즉시 나타난다. |

## 범위 밖

- 번호 생성 알고리즘, 가중치, 고정 번호 규칙, 실데이터 갱신 방식의 변경
- 추첨 효과음/배경음의 자산·타이밍·저장 형식 변경
- 새 서버, 분석 도구, 로그인, 결제, 광고, 외부 디자인 시스템 또는 이미지 생성 도입
- GitHub Pages 배포 설정, 하네스 전면 이관, CI/배포 게이트 변경 (별도 High-Risk 작업)
- 당첨금/출현 통계의 데이터 의미나 표기 기준 변경

## 데이터·외부 인터페이스·안전 경계

- 입력 데이터: 기존 `src/data/draws.json`, `src/data/latestPrize.json`만 사용한다. 데이터 포맷과 갱신 주기는 바꾸지 않는다.
- 브라우저 저장소: 기존 소리 설정 저장 동작만 유지하며, 새 사용자 식별 정보·추적 데이터·개인정보를 저장하지 않는다.
- 외부 호출: 새 네트워크 요청·유료 호출은 추가하지 않는다.
- 데이터 손실/되돌리기: UI 소스와 테스트만 변경 대상으로 하며, 문제 시 해당 UI 변경 커밋을 되돌리는 방식으로 복구한다.

## 미해결 질문 및 검증 계획

- 방향·범위·완료 조건·Standard 위험등급은 2026-07-15 사용자 승인을 받았다. 구현 전 독립 설계와 Work Package를 확정한다.
- 현재 작업 환경에는 `npm` 실행 파일과 `node_modules`가 없어 기존 검증 명령을 실행하지 못했다. 구현 전 의존성 설치 권한/방법을 확인하고, lint·typecheck·test·build를 실제로 재검증한다.
- 실제 GitHub Pages 공개 갱신은 코드 검토·검증과 별개의 production 실행 승인으로 다룬다.

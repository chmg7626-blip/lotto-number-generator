# 모바일 기본 웹 UI — Codex 독립 설계안

- 작성일: 2026-07-15
- 기준 spec: docs/specs/mobile-draw-stage-ui.md
- 상태: 결정됨 (2026-07-15 사용자 확정)
- 입력 범위: Approved SPEC, 현재 `src/App.tsx`, `src/components/GeneratorPanel.tsx`, 관련 컴포넌트와 CSS. 기존 구현 대화나 다른 AI의 설계안은 입력으로 사용하지 않았다.

## 구조

기존 도메인·사운드·오버레이 상태 흐름은 그대로 두고, 페이지의 JSX 배치와 CSS만 다음처럼 재구성한다.

```text
app-content
├─ pagebg (저강도 배경 장식)
├─ DisclaimerBanner
├─ header.site-header
│  ├─ logo
│  └─ home-sound-toggle
├─ main
│  ├─ section.intro (짧은 안내)
│  ├─ WinningBar (최신 당첨번호)
│  ├─ GeneratorPanel
│  │  ├─ section.generator-card
│  │  │  ├─ 생성 모드
│  │  │  ├─ 행운수 번호판 (fixed일 때만)
│  │  │  ├─ 게임 수
│  │  │  └─ 번호 뽑기
│  │  └─ Ticket (결과가 있을 때만)
│  ├─ 당첨금액
│  └─ 출현 통계
└─ SourceNotice
```

- `App.tsx`가 사이트 수준의 `header`와 `main` 순서를 맡는다. `handleDraw`, `pendingDraw`, reduced-motion 분기, BGM 제어, `DrawOverlay` 연결은 변경하지 않는다.
- `GeneratorPanel.tsx`는 번호 생성 제어와 결과만 맡는다. 현재의 모드·횟수·고정 번호 상태와 `chip`, `cnt`, `drawbtn` 클래스 계약은 유지한다.
- 정적 장식인 `LottoMachine`은 일반 페이지에서 제거한다. 실제 추첨 연출은 `DrawOverlay`의 canvas이므로 제거해도 생성·사운드·결과 확정 동작에 영향을 주지 않는다.
- `WinningBar`, `NumberPad`, `Ticket`은 로직 변경 없이 재사용한다. 필요하면 클래스와 CSS만 조정한다.

## 핵심 결정

1. **일반 웹 정보 구조를 우선한다.**
   헤더와 소개를 일반 문서처럼 배치하고, 최신 당첨정보와 생성 카드를 분리한다. 첫 화면에서 사용자는 어디서 번호를 생성하는지 바로 알아볼 수 있고, 통계/당첨금은 그 뒤의 보조 정보가 된다.

2. **로또 정체성은 포인트로 제한한다.**
   남색은 배경, 금색은 주요 CTA·선택 상태, 번호 공은 최신 결과와 생성 결과의 강조에만 쓴다. 큰 글로우·별·정적 추첨기처럼 행동을 방해하는 장식은 줄인다.

3. **마크업 이동만으로 상단 구조를 정상화한다.**
   로고·소리 토글·소개 문구를 `GeneratorPanel`에서 `App`의 사이트 구조로 옮긴다. 새 UI 라이브러리나 공용 디자인 시스템을 만들지 않고, 이미 있는 상태와 이벤트 핸들러를 그대로 연결한다.

4. **반응형은 작은 폭을 기준으로 명시한다.**
   320px에서 최신 당첨번호 공은 줄바꿈할 수 있게 하고, 로고/소리 토글은 절대 위치를 쓰지 않는다. 모드칩은 같은 폭의 그리드 또는 안전한 줄바꿈을 사용하고, 번호판·통계 그리드는 모바일에서 5열로 줄인다. 공 크기와 간격도 함께 낮춘다.

5. **현재 동작 계약을 보호한다.**
   `행운수 고정`의 0~5개 제한·`aria-pressed`·`disabled`, `×1~×5`, 데이터 없음 랜덤 폴백, 결과 확인 전 용지 미반영, reduced-motion 즉시 결과, 소리 저장 및 오버레이 focus/inert 계약은 변경하지 않는다.

## 장점

- 현재 앱의 핵심 가치인 번호 생성이 바로 보이고, 사용자가 이미 아는 웹 사용 방식과 일치한다.
- CSS 중심 변경이라 도메인·데이터·사운드 회귀 면적이 작다.
- 320px~390px의 실제 문제(당첨 공 띠, 절대 위치 헤더, 9열 번호판/통계)를 각각 명시적으로 해소한다.
- 기존 테스트의 핵심 selector와 도메인 인터페이스를 유지해 검증 비용을 낮춘다.

## 위험과 대응

| 위험 | 대응 |
| --- | --- |
| JSX 이동 중 소리 토글 또는 자동재생 첫 클릭 흐름이 달라질 수 있음 | `home-sound-toggle`의 이벤트와 `App`의 document capture listener를 그대로 보존하고 기존 BGM 테스트를 실행한다. |
| CSS만으로는 실제 뷰포트의 잘림을 놓칠 수 있음 | 320px, 390px, 768px, 1280px에서 수평 스크롤, 버튼 겹침, 번호판/통계 공의 잘림을 브라우저로 확인한다. |
| 정적 추첨기를 없애면 로또 느낌이 약해질 수 있음 | 로고·번호 공·선택 상태·결과 카드에서 색과 공을 유지하고, 실제 추첨 오버레이는 그대로 둔다. |
| 결과 카드의 공/복사 버튼이 좁은 폭에서 한 줄을 넘을 수 있음 | `Ticket`의 기존 클래스에 줄바꿈과 폭 제한 규칙을 추가하고, 결과를 만든 후 작은 폭으로 확인한다. |
| 배포 화면을 바로 바꾸면 되돌리기 비용이 생김 | 코드 검토와 로컬 검증을 마친 변경만 별도 커밋으로 만들며, 공개 갱신은 별도 승인 후에만 한다. |

## 구현 및 검증 순서

1. `App.tsx`에 일반 웹 header/main 골격과 소개 영역을 배치하고, `GeneratorPanel`에서 사이트 상단 요소와 정적 추첨기를 제거한다. 기존 추첨·소리 테스트를 우선 실행한다.
2. `styles.css`의 상단·생성 영역을 카드/섹션 기반으로 바꾸고, 320px/390px 반응형 규칙을 추가한다. 기존 토큰과 클래스는 가능한 유지한다.
3. `WinningBar`, `NumberPad`, `Ticket`, 통계 그리드의 좁은 폭 규칙을 보완한다. 320px에서 horizontal overflow가 없는지 확인한다.
4. 필요한 DOM/접근성 회귀 테스트를 보강하고 `lint`, `typecheck`, `test`, `build`를 실행한다.
5. 로컬 브라우저에서 320px·390px·768px·1280px 및 reduced-motion을 확인한다. 결과·sound·fixed mode·×5를 교차 점검한다.
6. Standard 검증 및 독립 리뷰 준비 상태를 확인한 뒤, 실제 공개 배포는 별도 승인으로 다룬다.

## 목적과 사용자 작업

첫 화면에서 사용자가 해야 할 한 가지 일은 **생성 방식을 고르고 번호를 뽑는 것**이다. B안은 결과를 기다리는 방송 화면보다, 손에 쥔 로또 용지를 정리하는 도구처럼 보이게 한다. 모든 생성 규칙·사운드·오버레이·데이터 고지·통계의 의미는 A안과 동일하게 유지한다.

## Creative Direction

- 감정: 과장된 행운 연출보다 차분하고 신뢰할 수 있는 "내 번호 정리" 경험
- 정보 밀도: 중저 — 첫 화면의 불필요한 여백을 줄이되 제어 문구와 고지는 읽을 수 있게 유지
- 움직임: 정적 중심 — 기존 추첨 오버레이만 표현적이고, 일반 화면에 새 장식 모션을 추가하지 않음
- 레이아웃: 익숙한 도구형 — 모바일과 넓은 화면 모두 현재의 단일 순차 흐름을 유지하고, 생성 데스크의 대비와 여백으로 작업을 분리
- 언어 톤: 기능적·친근함 — 기존 한국어 라벨을 유지하고 예측·당첨 보장처럼 오해될 문구를 추가하지 않음

## 유지할 요소

- 실제 로또 구간색 번호 공, 남색 잉크와 금색 CTA, 기존 마스코트
- `헤더 → 소개 → 최신 당첨번호 → 생성 → 결과 → 정보 → 출처`의 DOM 순서와 모든 현재 버튼/ARIA 계약
- 320px 단일 열, 5열 번호판·통계, reduced-motion, 오버레이 확인 뒤 결과 확정

## 피할 요소

- 네온·별·방송 세트가 작업 내용을 앞서는 무대형 장식
- 의미 없는 glass 카드의 중첩, 과도한 그림자, 색으로만 표현하는 선택 상태
- 새 폰트·아이콘·이미지·UI 라이브러리 또는 외부 요청

## 구조와 화면 상태

```text
mobile and desktop
  compact header → compact intro → latest ticket → bright generator desk
  → result ticket → dark supporting information → source
```

- 기본: 기본 모드와 `×1`이 선택된 밝은 생성 카드, 최신 회차 정보가 모두 보인다.
- 고정 번호: 기존 0~5개 선택 번호판은 카드 안에서 넓게 유지하며, 320px에서도 공은 38px로 유지하되 클릭/탭 대상은 44px 이상으로 확보한다. 선택/비활성은 테두리·명도·`aria-pressed`로 함께 전달한다.
- 성공: 기존 오버레이 확인 후 결과 용지가 생성 카드 바로 아래에 나온다.
- 빈 데이터/비활성: 기존 `데이터 없음` 및 전체 해제 disabled 상태의 문구·DOM 계약을 유지한다.

## 반응형·접근성·성능 제약

- 320·390px: 최소 44px 제어, 수평 스크롤 없음, 길어진 한국어 라벨은 줄바꿈 가능해야 한다.
- 768·1280px: 생성 패널 폭은 과도하게 넓히지 않고, 밝은 생성 데스크가 남색 정보 섹션과 명확히 구분돼야 한다.
- focus outline, semantic group, `aria-pressed`, `disabled`, 기존 reduced-motion 계약을 보존하고, 추첨 dialog는 Tab/Shift+Tab이 내부 제어 사이에서 순환하게 한다.
- root modifier class와 현재 CSS/마크업만 사용하며 새 asset·dependency·네트워크 요청은 추가하지 않는다.

## 검증 방법

- 기존 lint, 두 TypeScript typecheck, Vitest, Vite production build를 실행한다.
- localhost에서 기본/고정 번호/결과 상태를 320·390·768·1280px로 실제 렌더해 overflow·제어 도달성·시각 위계를 확인한다.
- 실제 Tab/Enter/Space 수동 조작 및 원격 CI·독립 리뷰는 가능한 환경이 없으면 `UNVERIFIED`로 남긴다.

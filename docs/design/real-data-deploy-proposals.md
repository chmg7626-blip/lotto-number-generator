# 실데이터 교체와 GitHub Pages 배포 — 독립 설계안 비교

- 작성일: 2026-07-06
- 기준 spec: docs/specs/real-data-deploy.md
- 참고 research: docs/research/lotto-draw-data.md
- 상태: 결정됨

Claude Code와 Codex가 같은 spec(과 research)만 보고 서로 독립적으로 작성한 두 설계안이다.
두 안이 모두 나오기 전에는 서로의 설계안을 공유하지 않았다 (독립성). 사용자가 방향을 결정한다.

## Claude Code 설계안

- 구조:
  ```
  scripts/update-draws.mjs        # Node 내장 fetch — 새 의존성 0
  src/data/draws.json             # 전 회차 당첨번호 (스크립트 생성물)
  src/data/latestPrize.json       # 최신 회차 등수별 당첨금 (생성물, 분리)
  src/domain/drawsValidation.ts   # 스키마 검증 — 스크립트·앱 테스트 공용
  .github/workflows/ci.yml        # 기존 확장: main push 시 검사 통과 후 deploy job
  .github/workflows/update-data.yml # 주간 크론 갱신 (스크립트 실행→변경 시 커밋)
  vite.config.ts                  # base: '/lotto-number-generator/'
  ```
- 핵심 결정:
  1. **취득 스파이크를 첫 WP로**: 동행복권 엑셀 다운로드 직접 호출/회차 조회 응답 형식을
     구현 착수 전에 확인한다. 실패 대비로 스크립트에 `--from-file` 오프라인 모드(사용자가
     받은 파일을 파싱)를 폴백으로 내장 — 자동 갱신이 막혀도 수동 경로가 남는다.
  2. **증분 갱신**: 기존 draws.json의 최대 회차 이후만 요청(주간 실행 시 요청 1~2건 —
     차단·부하 위험 최소). 전체 재수집은 `--full` 옵션. 이미 최신이면 파일 무변경(멱등).
  3. **검증 모듈을 src/domain에 공용 배치**: 스크립트 실행 시 검증 실패 → 파일 미변경·exit 1.
     같은 모듈을 Vitest 테스트가 draws.json에 대해 실행(범위·중복·연속성).
  4. **자동 갱신 커밋은 main 직접**: update-data.yml이 `contents: write`로 봇 커밋.
     branch protection이 켜지면 봇 예외 설정을 사용자에게 안내.
  5. **배포는 ci.yml 확장 단일 파이프라인**: main push 시 lint·typecheck·test·build 통과 후
     같은 워크플로우의 deploy job(actions/deploy-pages)이 실행 — "검사 통과 후 배포"를
     워크플로우 구조로 보장, 파일 수 최소화.
  6. 당첨금은 같은 스크립트가 latestPrize.json으로 동시 갱신. PrizeTable에 금액 포맷
     유틸(+테스트) 추가.
- 장점: 새 의존성 0. 증분이라 요청 수 최소. 검증 로직 중복 없음(스크립트·테스트 공용).
  단일 CI 파이프라인이 배포 게이트를 구조적으로 보장. 취득 실패 시 오프라인 폴백으로
  제품이 멈추지 않음.
- 위험: 취득 엔드포인트 미확인(스파이크 실패 시 자동 갱신 범위 재논의 필요). 봇 main 직접
  커밋은 운영 규칙의 예외. 증분 로직 버그 시 중간 회차 누락 가능(연속성 검증으로 방어).
  크론 시각과 동행복권 데이터 게시 시각 불일치 가능.
- 구현 및 검증 순서: WP-1 취득 스파이크 → WP-2 검증 모듈+테스트 → WP-3 갱신 스크립트
  (증분+오프라인 폴백, fixture 테스트) → WP-4 전체 데이터 생성·앱 교체(샘플 import 제거·
  prize 교체·출처 문구) → WP-5 vite base+deploy job → WP-6 주간 크론 워크플로우 →
  로컬 검사(lint·typecheck·test·build) → codex-review.

## Codex 설계안

- 구조:
  - 데이터 갱신 스크립트: 동행복권 공식 로또 6/45 회차별 당첨번호의 **엑셀 다운로드**를 받아
    회차별 당첨번호와 최신 회차 당첨금 데이터로 변환.
  - 생성 데이터: 당첨번호 전체 회차 JSON과 최신 회차 당첨금 파일 **분리**. 당첨번호 레코드는
    기존 `{ round, date?, numbers[6], bonus }` 스키마 유지.
  - 검증 계층: 변환 결과를 파일에 쓰기 **전에** 회차 연속·번호 범위·중복·보너스·최신 당첨금
    구조를 검증.
  - 앱 통합: 가짜 샘플 import 제거, 생성 데이터만 사용, UI "샘플" 표기 제거.
  - 출처 표기: 앱 화면과 데이터 산출물에 출처·비공식·재배포 허가 아님·권리자 요청 시 삭제
    수용 문구.
  - 자동화: 주간 갱신 워크플로우와 Pages 배포 워크플로우를 분리하거나, 같은 워크플로우
    안에서 품질 게이트 이후 배포.
- 핵심 결정:
  1. **취득은 공식 엑셀 다운로드로 고정**. 구현 초기에 URL·응답 형식을 확인하고,
     스크립트화가 불가능하면 HTML 파싱으로 임의 전환하지 말고 **spec 재논의**가 필요하다.
  2. 스크립트는 **fetch → parse → normalize → validate → compare → atomic write** 순서.
     검증 실패·네트워크 실패 시 기존 파일을 건드리지 않고 0이 아닌 종료 코드로 실패.
  3. 당첨금은 최신 회차의 1~5등 `rank`, `winnerCount`, `prizePerGame`만 별도 파일.
     당첨금의 round는 당첨번호 최신 round와 **반드시 같아야** 한다.
  4. 주간 갱신은 `workflow_dispatch`와 함께 **토요일 15:30 UTC(일요일 00:30 KST) 이후**
     실행. 새 회차가 확정돼 있으면 커밋하고, 변경 없으면 커밋하지 않는다.
  5. main 직접 push 긴장은 **데이터 전용 봇 커밋으로 제한**: 워크플로우에서 **diff
     allowlist**를 걸어 생성 데이터 파일 외 변경이 생기면 실패시킨다.
  6. Pages 배포는 기존 lint·typecheck·test·build 통과 뒤에만 실행. vite `base`는
     `/lotto-number-generator/`, 오디오·데이터 URL은 `import.meta.env.BASE_URL` 또는
     번들 import로 하위 경로 깨짐 방지.
- 장점: 공식 출처 하나만 기준으로 삼아 데이터 불일치 가능성 낮음. 변환·검증 분리로
  네트워크 없이 fixture 기반 테스트 가능. 멱등 출력과 atomic write로 이미 최신이면 불필요한
  커밋 없음. 파일 분리로 로직·UI·파일 크기 관리 단순. 법적 리스크는 제거 못 하지만 명시
  문구로 사용자 결정의 완화 조치 충족.
- 위험: 엑셀 다운로드의 직접 호출 URL·응답 형식이 미확인 — 이 확인이 구현 첫 관문.
  전체 회차 JSON 재배포 리스크는 해소되지 않았고 삭제하더라도 공개 git 이력에 남는다는
  점은 별도 운영 리스크. 기존 `DrawsFile`에 metadata 필드를 허용하지 않는다면 "스키마 유지"와
  "데이터 파일 내 출처 표기"가 충돌 가능 — 레코드 스키마를 깨지 않는 쪽을 우선하고 별도
  metadata 파일이나 README 반영으로 spec 확인 필요. 공식 엑셀 컬럼명·레이아웃이 바뀌면
  갱신 워크플로우가 실패하고 데이터가 stale — 실패가 조용히 무시되면 안 된다. 봇의 main
  데이터 커밋은 운영 규칙 예외이므로 diff allowlist와 사전 테스트가 필요하다.
- 구현 및 검증 순서: ① README·샘플 import 위치 확인 → ② 엑셀 다운로드를 수동으로 확인해
  스크립트 처리 가능성 판정 → ③ 변환 모듈(입력: 엑셀 원시 데이터 / 출력: 전체 JSON+최신
  당첨금) → ④ 검증 모듈(연속·범위·중복·보너스·당첨금 구조) → ⑤ 파일 쓰기는 검증 통과 후,
  deterministic formatting으로 이미 최신이면 diff 없음 → ⑥ 앱에서 샘플 제거·실데이터
  import·"샘플" 표기 제거 → ⑦ 출처·비공식·삭제 수용 문구 연결, 금지어 테스트 통과 확인 →
  ⑧ 주간 갱신 워크플로우(실패는 드러나게, 변경 없음은 성공+커밋 없음) → ⑨ Pages 배포
  워크플로우를 품질 게이트 뒤에 연결, base·오디오·데이터 경로 검증 → ⑩ 최종:
  lint/typecheck/test/build + Pages URL에서 JS·CSS·오디오·데이터 로드 확인.

## 핵심 차이

| 항목                  | Claude Code                                   | Codex                                                               |
| --------------------- | --------------------------------------------- | ------------------------------------------------------------------- |
| 취득 실패 대비        | `--from-file` 오프라인 폴백을 스크립트에 내장 | 폴백 없이 고정 — 불가하면 임의 전환 말고 spec 재논의                |
| 갱신 방식             | 증분(최대 회차 이후만 요청, `--full` 옵션)    | 전체 파이프라인 fetch→…→compare→atomic write                        |
| 봇 main 커밋 통제     | `contents: write` + 필요 시 보호 예외 안내    | **diff allowlist** — 데이터 파일 외 변경 시 워크플로우 실패         |
| 워크플로우 구성       | ci.yml 확장 단일 파이프라인 + 갱신 워크플로우 | 갱신/배포 분리 허용, 크론 시각 명시(토 15:30 UTC)                   |
| 검증 모듈 위치        | src/domain 공용(스크립트·앱 테스트 공유)      | 변환 모듈과 검증 모듈 분리, fixture 테스트                          |
| 데이터 파일 출처 표기 | JSON meta 필드 또는 별도 파일(열어둠)         | 레코드 스키마 유지 우선 — 별도 metadata 파일/README, spec 확인 요청 |
| 당첨금 정합성         | 같은 스크립트가 동시 갱신(암묵 정합)          | prize.round == draws 최신 round **명시 검증**                       |

## 각 도구의 추천

- Claude Code 추천: **절충안** — A의 골격(공용 검증 모듈·증분 갱신·단일 CI 파이프라인)에
  B의 통제 장치(diff allowlist, prize-round 일치 명시 검증, atomic write, 크론 시각 명시)를
  결합. B의 "폴백 없이 spec 재논의"는 변경 통제 원칙에 더 부합하므로 오프라인 폴백은
  스파이크 결과를 보고 결정. 비용: 결합 항목만큼 구현 범위가 조금 늘어난다.
- Codex 추천: 자기 설계안(B) — 공식 엑셀 단일 소스 고정, 검증 통과 전 파일 미변경,
  diff allowlist로 봇 커밋 제한이 핵심.

## 사용자 결정

- 선택: 절충안 (2026-07-06)
- 선택 이유: A의 골격(공용 검증 모듈·증분 갱신·단일 CI 파이프라인)이 단순하고, B의 통제
  장치(diff allowlist·prize-round 명시 검증·atomic write·크론 시각·폴백 없이 spec 재논의)가
  변경 통제 원칙에 부합 — 두 안의 강점 결합.
- 추가 지시: 오프라인 폴백(`--from-file`)은 미리 만들지 않는다. 취득 스파이크(WP-1) 결과
  스크립트화가 불가능할 때만 spec 재논의로 다룬다.

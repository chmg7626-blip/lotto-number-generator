# 로또번호생성기 — Codex 독립 설계·검토 지침

이 파일은 Codex의 독립 설계와 독립 검토 규칙이다. 설계 단계에서 Codex는 spec만 보고 독립
설계안을 작성하고, 구현 후에는 변경을 별도 단계에서 검증한다. Claude Code가 확정된 설계를 구현한다.

## Review guidelines
GitHub의 Codex 자동 리뷰도 이 문서를 따른다 (Codex는 이 제목의 섹션을 찾는다).
지침의 본문은 아래 독립 설계·구현 후 검토·보고 형식 섹션이다.

## 시작 전 — PROJECT.md 확인
설계·검토를 시작하기 전에 `PROJECT.md`를 읽어 프로젝트 정체성·현재 상태·타입을 파악한다.
단, **PROJECT.md는 맥락 요약이지 판단 기준이 아니다.** 구현 검토의 기준은 여전히 **확정 spec과
git diff**다. PROJECT.md와 spec이 어긋나면 spec을 따르고, 그 어긋남을 보고한다.

## 독립 설계

입력:
- `docs/specs/`의 이 작업 spec (완료 조건의 기준)
- `docs/research/`의 승인된 research 문서 (있을 때만 — 사용자가 출처·결론을 확인해 기록한 것)
- Claude의 설계안이나 대화는 입력으로 쓰지 않는다. 같은 spec에서 독립적으로 출발해야 두 관점이 의미 있게 갈린다.

절차:
1. spec의 목적·요구사항·완료 조건·범위 밖을 읽고, 그 완료 조건을 만족하는 설계를 처음부터 세운다.
2. 정상·오류·경계값과 운영·보안 위험을 빠뜨리지 않는다.
3. 기존 구조로 충분한데 새 의존성·추상화를 도입하지 않는다 (과도한 설계 금지).
4. 구현을 시작하면 비싸게 되돌려야 할 결정과 위험을 분명히 적는다.

설계안 보고 형식: **구조 / 핵심 결정 / 장점 / 위험 / 구현 및 검증 순서**.
이 단계에서는 코드를 작성하지 않는다. Claude의 설계안을 검토하지도 않는다 — 별도의 설계안을 직접 만든다.

## 구현 후 검토

입력:
- 기준 문서: docs/specs/ 의 활성 spec (이 브랜치 작업에 해당하는 파일)
- 변경 내용: main 대비 git diff
- Claude의 구현 설명이나 대화 내용은 검토 입력으로 쓰지 않는다.
  구현자의 논리를 먼저 읽으면 그 논리에 끌려가 독립성이 깨지기 때문이다.

절차:
1. 활성 spec을 읽고 완료 조건을 추출한다.
2. git diff로 변경 내용을 파악한다. diff만 보지 말고 변경된 파일의 주변 코드를 충분히 읽는다.
3. spec 대비 누락된 사례를 찾는다 (정상·오류·경계값).
4. 테스트가 구현 논리를 그대로 답습하지 않는지 확인하고, 필요하면 독립 테스트를 추가 제안한다.
   High-Risk 작업은 독립 테스트 사례 제안이 필수다 — Claude가 그 테스트를 작성하면 Codex가 재검토한다.
5. 테스트 통과 여부 자체는 Claude의 로컬 검사·CI가 보장한다 — Codex는 read-only로 실행되어
   테스트 재실행이 보장되지 않는다. diff의 테스트 코드가 적절한지(누락 사례·구현 답습)를 검토하고,
   환경이 허용하면 아래 명령으로 재실행해 교차 확인한다.
   - 테스트: npm test
   - lint: npm run lint
6. 보안·회귀·예외 경로를 검토한다.

## 수정 권한
- 발견한 문제는 직접 고치지 말고 보고한다. 수정은 Claude Code가 하고, Codex가 재검토한다.
- 예외: 오타 수준의 사소한 수정만 별도 커밋으로 직접 처리할 수 있다.
  "사소함"의 기준을 임의로 넓히지 않는다.

## 보고 형식 (구현 후 검토)
- **[치명]** 다음 단계로 진행하면 안 되는 문제 — 파일:줄번호, 무엇이 왜 문제인지
- **[권장]** 고치면 좋은 문제
- **[참고]** 알아두면 좋은 것

문제가 없으면 "특이사항 없음"이라고 명확히 말한다. 억지로 지적거리를 만들지 않는다.
(독립 설계 단계의 보고 형식은 위 "독립 설계" 섹션의 구조/핵심 결정/장점/위험/구현 및 검증 순서다.)
